       now = time.monotonic()
       if (self._sort_keys is not None

       meta = self._col.find_one({"_id": META_ID})

**PERF-03: Redis bounds-check elimination**
The fix is in the facade layer. In `src/asebytes/io.py` ASEIO.__getitem__ (line 229), `src/asebytes/_object_io.py` ObjectIO.__getitem__, and `src/asebytes/_blob_io.py` BlobIO.__getitem__:

For ASEIO (io.py), replace the integer-index branch (lines 233-250) with:
```python
if isinstance(index, int):
    try:
        n = len(self)
    except TypeError:
        # Backend with unknown length (file-based, streaming).
        if index < 0:
            len(self)  # raises TypeError
        row = self._read_row(index)
        return self._build_result(row)
    if index < 0:
        index += n
    # Delegate bounds check to backend -- avoids redundant len() in
    # backends that check bounds internally (e.g. Redis Lua scripts).
    try:
        row = self._read_row(index)
    except IndexError:
        raise IndexError(index) from None
    return self._build_result(row)
```
The key change: remove `if index < 0 or index >= n: raise IndexError(index)` and instead catch IndexError from `_read_row`. The `len(self)` call is still needed for negative index normalization but not for bounds checking.

Apply the same pattern to ObjectIO.__getitem__ in `_object_io.py` (line 194) and BlobIO.__getitem__ in `_blob_io.py` (line 161). In ObjectIO and BlobIO, the pattern is simpler since there's no TypeError path:
```python
if isinstance(index, int):
    n = len(self)
    if index < 0:
        index += n
    try:
        return self._backend.get(index)
    except IndexError:
        raise IndexError(index) from None
```

IMPORTANT: For ASEIO, we still call `len(self)` once for negative index normalization. The optimization is that we do NOT call it again to pre-check bounds. For Redis, this means 1 LLEN (for len) + 1 Lua GET (which also does LLEN) = 2 RT total. But wait -- the current code also calls `len(self)` once (line 235) and then does bounds check. The real win is that for NON-negative indices, we can skip the len() call entirely. Let me reconsider.

Actually, the real optimization per RESEARCH.md: "Eliminate redundant len() call. The Lua script already does LLEN+bounds internally." The current code calls `n = len(self)` (1 LLEN RT) then `_read_row(index)` (1 Lua RT which does LLEN again). For positive indices, we don't need len() at all. For negative indices we still need it.

Revised approach for all three facades:
```python
if isinstance(index, int):
    if index < 0:
        n = len(self)
        index += n
        if index < 0:
            raise IndexError(index - n)
    try:
        row = self._read_row(index)  # or self._backend.get(index)
    except IndexError:
        raise IndexError(index) from None
    return self._build_result(row)  # or return result directly
```
This way, `len()` is only called for negative indices, saving 1 RT for Redis on positive index access (the common case).

For ASEIO specifically, keep the TypeError handling for unknown-length backends:
```python
if isinstance(index, int):
    if index < 0:
        try:
            n = len(self)
        except TypeError:
            len(self)  # re-raise TypeError
        index += n
        if index < 0:
            raise IndexError(index - n)
    try:
        row = self._read_row(index)
    except IndexError:
        raise IndexError(index) from None
    return self._build_result(row)
```
